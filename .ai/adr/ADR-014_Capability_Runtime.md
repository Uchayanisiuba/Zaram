# ADR-014: Capability Runtime
**Status:** Accepted
**Milestone:** 1.5
**Context:** With Milestones 1.0‚Äì1.4 verified (Kernel, Runtime Registry, Event Bus, Execution Engine, Embodiment Framework, Cognitive Runtime, World Runtime, Executive Runtime), the architecture has an Executive Runtime that is the single authority for high-level decision-making. But the executive has no structured way to discover *what Zaram can do* ‚Äî which capabilities (filesystem, browser, calculator, clipboard, email, vision, speech, camera, OCR, code execution, search, plugins, automation, future agents) exist, what permissions they require, where they execute, and their cost/latency. Without a capability layer, the executive would be tempted to call tools directly, violating subsystem independence and the Runtime Contract ("A Runtime may NEVER directly call or import another Runtime's implementation").
**Decision:** Introduce the `CapabilityRuntime` under `src/runtime/capability/` as the OS capability discovery and execution interface. It is the single source of truth for every capability available to the OS and exposes ONLY capability metadata (descriptors). It owns registration, O(1) lookup by id, indexed lookup by category, declarative filtering (category / permissions / availability / location / enabled), and resolution of the best candidate for a need. The Executive Runtime is injected with the `ICapabilityRuntime` interface and requests capabilities through it ‚Äî it never calls tools directly and never imports a concrete capability implementation. The actual execution remains the responsibility of the separate Execution Engine (not built in M1.5). The runtime reuses the existing DI container (`TOKENS.capabilityRuntime`, singleton) and the existing `Set`-based pub/sub pattern; it introduces no timers, polling, or `requestAnimationFrame`. It must NOT import the drawing layer, the body layer, concrete avatars, the character projection, the animation engine, frame snapshots, the orb drawing code, the desktop shell, any GPU/3D engine, or the Emotion/Behaviour/Presence/Character/Embodiment runtimes.
**Consequences:**
- A single, queryable catalogue of OS capabilities exists, decoupled from execution. The Executive can plan against metadata (permissions, location, cost, latency) without coupling to implementations.
- Direction of coupling is correct: the Executive depends on `ICapabilityRuntime` (interface), never the concrete runtime or a tool. Subsystem Independence and the Runtime Contract are preserved.
- 100% drawing-layer and body-layer independence is preserved; the Capability Runtime emits no `FrameState` and no `CharacterFrame`.
- `CharacterFrame`, `FrameState`, the Embodiment Framework, `AnimationRuntime`, the renderer pipeline, `OrbRenderer`, `LivingOrbAdapter`, and all M1.0‚ÄìM1.4 contracts remain byte-for-byte unchanged except for the additive DI registration of `capabilityRuntime` and the additive, interface-only injection into the Executive Runtime.
**Compliance:**
- No `setInterval`/`setTimeout`/`requestAnimationFrame`/`while` in `src/runtime/capability/*`.
- Renderer/embodiment/character-pipeline independence asserted structurally by `tests/runtime/capability/capability-runtime.test.ts`.
- DI and Executive integration verified by the same test file and `bootstrapPresence()`.
- Performance verified by `tests/runtime/capability/capability-bench.test.ts` (10k registrations < 250ms; O(1) lookup < 0.01ms).
