# ADR-015: Execution Runtime
**Status:** Accepted
**Milestone:** 1.6
**Context:** With Milestones 1.0–1.5 verified (Kernel, Runtime Registry, Event Bus, Execution Engine, Embodiment Framework, Cognitive Runtime, World Runtime, Executive Runtime, Capability Runtime), the architecture has an Executive Runtime that decides what the AI should do and a Capability Runtime that discovers *what Zaram can do* — but no runtime that actually *executes* capabilities. The Capability Runtime explicitly exposes metadata only. The Executive Runtime explicitly never executes. Without a dedicated Execution Runtime, either the Capability Runtime would be forced to execute (violating its metadata-only contract) or the Executive Runtime would call tools directly (violating the single-authority decision-making contract and the Runtime Contract). The architecture requires a clear separation: discovery (Capability), decision (Executive), and execution (Execution).
**Decision:** Introduce `ExecutionRuntime` under `src/runtime/execution/` as the ONLY runtime allowed to invoke capabilities. It owns the complete execution lifecycle (Queued → Preparing → Running → Waiting → Retrying → Completed / Cancelled / Failed / RolledBack), timeout enforcement, retry logic, cancellation, progress reporting, execution history, audit trail, rollback hooks, permission enforcement, execution context, execution ids, and a deterministic state machine. It reuses the existing 30Hz scheduler (`PresenceRuntime.tick()`) for lifecycle advancement — no new timers, polling loops, or `requestAnimationFrame`. It is wired via the existing DI container as singleton `executionRuntime`, consumed by `PresenceRuntime` via DI only, and never imports the drawing layer, body layer, character projection, animation engine, frame snapshots, orb drawing code, desktop shell, or any GPU/3D engine. The Executive Runtime calls `executionRuntime.execute(request)` exclusively; it never calls capabilities directly.
**Consequences:**
- A single, testable execution surface enforces all cross-cutting concerns (timeout, retry, cancellation, permissions, audit) in one place. Capability implementations stay simple and focused on their core logic.
- Direction of coupling is correct: the Executive decides, the Execution Runtime executes. The Capability Runtime remains metadata-only. Subsystem Independence and the Runtime Contract are preserved.
- 100% drawing-layer and body-layer independence is preserved; the Execution Runtime emits no `FrameState` and no `CharacterFrame`.
- `CharacterFrame`, `FrameState`, the Embodiment Framework, `AnimationRuntime`, the renderer pipeline, `OrbRenderer`, `LivingOrbAdapter`, and all M1.0–M1.5 contracts remain byte-for-byte unchanged except for the additive DI registration of `executionRuntime` and the single `update(dt)` call in the reused tick.
**Compliance:**
- No `setInterval`/`setTimeout`/`requestAnimationFrame`/`while` in `src/runtime/execution/*`.
- Renderer/embodiment/character-pipeline independence asserted structurally by `tests/runtime/execution/execution-runtime.test.ts`.
- DI and PresenceRuntime integration verified by the same test file and `bootstrapPresence()`.
- Performance verified by `tests/runtime/execution/execution-runtime.test.ts` (10,000 executions < 15s; lifecycle transitions < 1ms).
